## Краткое ТЗ (сводка)

Источники: `Assets/Documentation/GDD.txt`, макеты `Assets/Documentation/Screens/Maket_*.png`, устные уточнения в переписке.

### 1. Продукт

| Параметр | Значение |
|----------|-----------|
| Рабочее название | The Last Tower Defence (репозиторий/проект) |
| Жанр | Tower Defense + Idle RPG |
| Платформа | Браузер (ПК и мобильные), **landscape**; **релизный билд — Unity WebGL** (см. п. 8 и заметки по окружению) |
| Сеттинг | Дарк-фэнтези |
| Управление в бою | Практически **zero-player**: игрок готовит партию между волнами; бой идёт автоматически |

### 2. Бой и мир

- Камера и читаемость: **вид сверху**, вертикальная дорожка/«коридор», **база внизу экрана**, противники движутся к базе (согласовано с GDD: защита единственной башни/базы).
- **Одна полоска HP базы** (на макете 1000 / 1000). Пустая полоска — поражение раунда (см. п. 5).
- Три героя в бою: **рыцарь** — ближний бой на дорожке; **лучник** — дальний бой с базы; **клерик** (в арте и UI менеджмента — **маг**) — поддержка и лечение (роль из GDD).
- С убитых врагов с **вероятностью** на землю падает **сундук** (визуал лута на макетах). После **победы во волне** все оставшиеся на поле сундуки **автоматически** забираются в экономику/инвентарь перед следующими экранами.

### 3. Структура волн (GDD)

- Волны **1–2**: обычные противники.
- Волна **3**: **босс** или группа элиты — конец цикла из трёх волн.
- После победы в **3-й волне** — **дирижабль-магазин**: редкое снаряжение за **твёрдую валюту** (hard currency).

### 4. Цикл между волнами (макеты + GDD)

Последовательность ключевых состояний:

1. **Бой (Maket_1 / Maket_5)** — волна N; дроп сундуков с шансом.
2. **Поле после волны (Maket_2)** — автосбор сундуков с земли.
3. **Провидица (Maket_3)** — лор и **данные о следующей волне**; кнопка **«Далее»** (на макете NEXT) — переход к менеджменту. Дополнительно: переключение/режимы иконками диалога и «i» (детали реализации — на усмотрение).
4. **Менеджмент (Maket_4 и варианты вкладок)** — экипировка и прогресс:
   - слева **переключатель героев**: рыцарь / лучник / маг (клерик);
   - **слева в окне** — слоты **экипировки и персональный контент** выбранного героя;
   - **справа** — **единая сетка инвентаря** на всех троих; сюда попадает собранный после волны лут;
   - внизу **четыре вкладки**: инвентарь, характеристики (в т.ч. распределение свободных очков), статистика (производные параметры), информация (текст);
   - **«Далее»** — старт **следующей волны** (снова экран боя).

Между циклами GDD дополнительно требует: распределение **XP**, фаза **торговли** (дирижабль каждые 3 волны после босса). В ТЗ это остаётся обязательной частью экономики, даже если отдельного макета дирижабля в папке Screens пока нет.

### 5. Поражение и «Гнев Мага» (GDD)

- Триггер: **HP базы = 0**.
- **Ульта мага**: все враги на поле уничтожаются.
- **Штраф:** весь лут текущей волны, **лежащий на земле**, сгорает.
- **Сохраняется:** надетая экипировка и **XP**, накопленный в текущем бою.
- Игрок **повторяет ту же волну**, став сильнее за счёт XP.

### 6. Инвентарь, лут и экономика (GDD)

- С убитых врагов — предметы (**оружие, броня, аксессуары**); визуально на макетах — сундуки как контейнер/метафора дропа — реализация должна согласовать тип предмета при открытии/сборе.
- **Общий инвентарь** (как на макете — сетка) + **индивидуальные слоты** экипировки у каждого героя.
- При переходе к следующей волне из фазы менеджмента (**«В бой»** / аналог): все предметы в «сумке», **не надетые на героев**, **автоматически продаются** за золото (**чистый инвентарь**).

### 7. Интерфейс и техтребования

- **Адаптивный UI** под разные разрешения, сохранение **коридорной** компоновки боя (GDD); модальные экраны — в стиле макетов (тёмная панель, декоративная рамка).
- **Лёгкий вес** сборки для браузера: упор на спрайты / атлас или Spine по возможности.
- **Сохранение:** автосейв прогресса (уровни героев, экипировка и т.д.) — **LocalStorage** и/или **сервер** (GDD).

### 8. WebGL как цель с первого дня

- Разработка в основном в **Editor**, но **регулярно** собирать и открывать **WebGL** (хотя бы после крупных задач): ловятся только в билде ограничения памяти, шейдеры, ввод, время первой загрузки.
- **Память и GC:** в бою и при прокачке избегать лишних аллокаций в кадр (кеш, пулы, осторожно с LINQ в горячих путях); статы и формулы — стабильные структуры, без «мусора» на каждый пересчёт.
- **Размер и загрузка:** минимальная первая сцена; тяжёлый контент — позже **Addressables** / потоковая подгрузка; сжатие текстур и аудио под веб.
- **Сейвы:** `PlayerPrefs` ≈ LocalStorage с **лимитом объёма**; при росте сохранений закладывать **IndexedDB** (нативный плагин `.jslib`) или **сервер**.
- **Ограничения среды:** нет полноценной многопоточности как на десктопе; часть .NET API недоступна — проверять под WebGL при выборе библиотек (сеть, файлы, потоки).
- **IL2CPP / stripping:** при рефлексии и сериализации — явный **`link.xml`**, иначе обрезание кода в релизе.

### 9. Вне скоупа краткого ТЗ

Детальная балансировка чисел, таблицы врагов, API сервера, полный список статов — по мере появления отдельных документов или задач.

## Заметки по окружению (по мере накопления)

- **Целевая платформа:** Unity **WebGL** с первого дня (не только «потом порт»). Имеет смысл держать в CI или в чек-листе релиза: периодическая **Development** и **Release** сборка WebGL + ручной прогон в Chrome/Edge и на одном мобильном браузере.
- **Статы и формулы:** горячий путь боя лучше профилировать с мыслью о **браузерном потолке памяти** и стоимости GC; тяжёлые пересчёты (полный пересчёт всех производных статов) — по событию (смена экипировки, баффа), а не каждый кадр без нужды.
- **Input System:** в **Project Settings → Player → Active Input Handling** выбрать **Input System Package** (или **Both** на время миграции, если ещё есть код на `UnityEngine.Input`). Для **EventSystem** на Canvas с новым стеком нужен компонент **Input System UI Input Module** вместо устаревшего Standalone Input Module.
